# CLI Command Contract Inventory

Status: migration gate for parser, grouping, and renderer changes.

This inventory records the current `agent-tracker` command-line compatibility
surface before any broader parser migration. Treat it as the baseline contract
for commands, aliases, arguments, options, JSON roots, stream behavior, exit
codes, environment defaults, shell completion, and command-boundary parsing.

## Migration Gate

Before moving additional commands away from the current root parser, a change
must preserve or deliberately version these contracts:

- existing flat commands remain available and keep their current option names;
- grouped commands are additive aliases unless a documented deprecation window
  exists;
- `--json` payload roots stay stable and do not inherit human formatting;
- human output stays copy-paste-safe and keeps diagnostics off stdout when a
  command is mutating state;
- common errors return non-zero without tracebacks;
- shell-completion helper flags stay hidden until completion support is
  explicitly designed;
- parser boundaries such as `launch-worker --command ...` keep all following
  tokens inside the worker command argv.

## Parser Boundary

The root parser is `argparse` and owns every flat command. The only runtime
grouped family is `agent-tracker intake`, which `main()` dispatches before root
`argparse` parsing to a small Typer/Click adapter. `build_parser()` still
defines an argparse `intake` subtree for parser introspection and compatibility
tests, but normal CLI execution uses the Typer adapter. Root help remains
generated by `argparse`; grouped intake help is generated by Typer with
completion helpers disabled.

The root command set is required. Running without a command is an argparse usage
error. Command handlers return integer process codes through `main()`.
Argparse validation errors and help exits still raise `SystemExit` when `main()`
is called directly in process, matching the current test style.

## Common Options And Config Precedence

Every project command except `init-project` accepts:

| Option | Contract |
| --- | --- |
| `--config` | Project config JSON path. Empty value falls back to `AGENT_TRACKER_CONFIG`. |
| `--db` | SQLite database override. Empty value falls back to `AGENT_TRACKER_DB`. |

Explicit CLI options take precedence over environment defaults. If neither
`--config` nor `AGENT_TRACKER_CONFIG` is provided, configuration loading fails
with an actionable `error: ...` message on stderr.

When `canonical_config_path` is configured, mutating commands must use the
canonical config path and must not override the configured canonical database.
Those refusals are command errors, not tracebacks.

Human renderer commands may accept `--no-color`. That option affects only human
rendering; JSON payloads and plain grep-friendly output are unchanged.

## Grouped Intake Aliases

Grouped intake commands are compatibility aliases for the flat intake commands.
They must remain additive and must not replace the flat commands.

| Grouped command | Flat command | Contract |
| --- | --- | --- |
| `intake record` | `record-intake` | Same intake JSON object, metadata merging, path report, and audit behavior. |
| `intake capture` | `capture-intake` | Same structured intake JSON object and required context fields. |
| `intake list` | `list-intake` | Same filters, human output, JSON root, and `--no-color` behavior. |
| `intake update` | `update-intake` | Same status update JSON object and actor behavior. |

Grouped intake accepts `--config`, `--db`, and `--no-color` at group level.
Leaf commands also accept `--config` and `--db`; leaf values override empty
group-level values. `intake list` accepts `--no-color` either at group level or
leaf level.

## Stdout And Stderr

Stdout is for successful human output, plain output, or JSON payloads. Stderr is
for command diagnostics and errors.

Mutating commands that load a project coordinator generally print a resolved
path report to stderr before changing state. This includes config path, state
root, task source root, database path, and task source path when available.
That report must not be mixed into JSON stdout.

`export-pr-notifications` is an explicit stream-ordering exception. JSON mode
prints only the JSON payload to stdout and does not emit the resolved path
report to stderr. Human mode computes the payload first, then prints the path
report to stderr before rendering the human export summary. Preserve that
ordering unless a later task deliberately versions the command contract.

Read-only JSON commands should keep stdout parseable as JSON and avoid path
reports on stderr unless the command already documents one. Human-rendered
commands may use restrained color, but output must remain meaningful when color
is disabled or stripped and must not use box-drawing characters.

## Exit Codes

The normal success code is `0`.

Known non-zero conventions:

- caught `FileNotFoundError`, `ImportError`, `json.JSONDecodeError`,
  `KeyError`, `sqlite3.Error`, and `ValueError` paths return `1` through
  `main()` and print `error: ...` to stderr without a traceback;
- `check-completion-integrity` returns `1` when integrity payload `ok` is false;
- `export-pr-notifications` returns `1` when export payload `ok` is false;
- argparse parse failures return `2` through `SystemExit`;
- help requests return `0` through `SystemExit` for direct in-process calls;
- grouped Typer intake validation follows Click/Typer exit codes, while
  handler errors are still surfaced through the shared command handlers.

## Shell Completion Policy

The CLI does not currently expose shell-completion installer or display helper
flags. Root argparse help must not include `install-completion` or
`show-completion`. Grouped Typer intake help also hides those helpers through
`add_completion=False`.

Adding shell completion is a separate compatibility project. It must document
the supported shells, install/display commands, generated command names, and
failure behavior before exposing helper flags.

## Command-Boundary Behavior

`launch-worker --command` uses argparse remainder parsing. Every token after
`--command` belongs to the worker command argv, even if the token looks like an
`agent-tracker` option. For example, a trailing `--json` after `--command` is
part of the worker argv and must not turn on `launch-worker --json`.

`launch-worker --command-string` is parsed with shell-like splitting through
`shlex.split`. When both `--command-string` and `--command` are supplied, the
command string takes precedence in the current handler.

## Required Regression Coverage Before Migration

The current test suite already covers representative JSON roots, error output,
environment precedence, flat and grouped intake parity, human no-box output,
`--no-color`, and `launch-worker --command` remainder behavior.

Before parser ownership changes, keep or expand tests that prove:

- root help includes flat commands and remains no-box;
- root and grouped help do not expose completion helpers;
- `release-lease` remains an alias for `release`;
- flat intake commands and grouped intake aliases keep matching JSON and human
  behavior;
- representative JSON roots remain stable for `status`, `overview`, `next`,
  intake, proposals, interventions, PR notifications, and worker launch;
- common service and validation errors are concise stderr messages without
  tracebacks;
- explicit `--config` and `--db` options continue to override environment
  defaults;
- `launch-worker --command` remains a hard parser boundary.

## Follow-Up Migration Work

This inventory is intentionally descriptive. It does not approve command
renames, parser migration, option deprecation, or completion support by itself.
Follow-up implementation tasks should cite this document as their baseline and
list every intentional contract change in their task contract and release notes.
